# 需求收集（未结构化）

## 候选方向

| 方向 | 核心形态 | 优点 | 缺点 | 代价 | 决策 |
|---|---|---|---|---|---|
| A | 纯对话式——Agent 在对话中自主规划、搜索、输出报告 | 交互极简，无学习成本 | 过程不透明，中途难干预，输出以线性文本为主 | 低 | ❌ 暂缓 |
| B | 结构化工作台——展示研究计划→用户确认/修改→可视化执行每一步→结构化报告+来源 | 过程透明可干预，可控性强，输出质量高 | UI 复杂度高，开发周期长 | 中-高 | ✅ 选定 |
| C | 两步半自动——对话中生成大纲→用户确认→Agent 后台异步执行→返回报告 | 兼顾简洁与可控 | 异步模式无实时反馈，等待感强 | 中 | ❌ 暂缓 |

## 选定方向：B — 结构化工作台

**取舍理由**：用户需要在研究过程中有可见、可控的参与节点（尤其是确认研究计划前可多轮交互澄清），纯对话式无法满足，异步模式缺少实时反馈。结构化工作台虽然 UI 复杂度更高，但对"深度研究"场景的质量和用户信任感至关重要。

## 澄清结果

| # | 问题 | 答案 |
|---|---|---|
| Q1 | 用户量预期 | 生产级，用户量过万 |
| Q2 | 平台 | Web 端 |
| Q3 | 离线支持 | 不需要 |
| Q4 | 搜索源接入方式 | 通过 MCP（Model Context Protocol）接入 |
| Q5 | 输出格式 | Markdown 报告，内容包含：①研究计划 ②各 Sub-agent 的研究结果 ③ Sub-agent 研究结果汇总 |
| Q6 | 是否可编辑报告 | 不能，报告为只读产物 |
| Q7 | 单次研究深度上限 | 主 Agent 最多拆分 **3-5 个 Sub-agent**；每个 Sub-agent 最多 **2 轮循环**（V1 硬限制） |
| Q8 | 历史持久化 | 是，研究历史需要持久化保存 |
| Q9 | 账号体系 | 需要（登录/注册） |
| Q10 | 隐私合规 | 暂不考虑 |
| Q11 | 自定义研究模板 | V1 提供固定模板枚举（技术调研/竞品分析/论文综述/自定义），用户新建研究时手动选择，不做自定义模板编辑 |
| Q12 | 多轮交互节点 | 仅在确认研究计划前支持多轮交互澄清/修改；计划确认后全自动执行 |

## 关键推断（基于 Q4 + Q5 的架构暗示）

> 以下为需求澄清过程中自然浮现的架构推论，非需求本身，供后续 Phase 参考。

- **MCP 为标准数据接入协议**：所有搜索源、数据源通过 MCP 标准化接入，意味着后端需实现 MCP Client 或复用社区 MCP SDK
- **Sub-agent 架构**：系统由一个主 Agent（编排研究计划）和多个 Sub-agent（并行执行具体研究子任务）组成，主 Agent 负责最终汇总
- **计划审批节点**：研究流程为 `输入主题 → 生成研究计划 → 多轮澄清/修改 → 用户确认 → Sub-agent 并行执行 → 汇总 → 输出报告`
