# PRD：自动化深度研究智能体（DeepResearch Agent）

## 1. 概述

- **目标**：为知识工作者提供一个基于结构化工作台的深度研究助手，将 1-2 小时的研究工作压缩到 5-10 分钟，输出系统化的 Markdown 研究报告。
- **背景**：当前信息搜索存在信息过载、缺少结构、重复劳动三大痛点。传统搜索引擎返回碎片化结果，用户需手动完成"搜索→阅读→总结→整理"全链路，过程耗时且不可复用。
- **非目标**：
  - 不做实时新闻/热点追踪
  - 不做通用闲聊式问答
  - 不替代文献管理工具（如 Zotero）
  - 不生成学术论文格式
  - 不做多语言翻译/本地化
  - v1 不做团队协作/分享

---

## 2. 用户角色

| 角色 | 描述 | 权限 |
|---|---|---|
| 匿名用户 | 未登录访问者 | 仅可登录/注册 |
| 普通用户 | 已注册登录的研究者 | 新建研究、管理自己的研究历史、查看报告 |
| 管理员 | 系统运维 | 用户管理、系统监控（v1 可简化为 CLI 操作） |

---

## 3. 核心用例

### UC-001: 用户注册

- **角色**: 匿名用户
- **前置条件**: 无
- **主流程**:
  1. 用户填写账号和密码
  2. 系统校验格式、密码强度
  3. 系统创建账号并返回成功
  4. 自动登录，跳转到主界面
- **替代流程**: 账号已存在 → 提示"该账号已被注册，请直接登录"
- **异常流程**: 输入校验失败 → 返回具体错误提示（密码过短 / 账号格式错误）

### UC-002: 用户登录

- **角色**: 匿名用户
- **前置条件**: 已注册
- **主流程**:
  1. 用户输入账号和密码
  2. 系统验证凭据
  3. 返回 JWT Token，前端存储并跳转主界面
- **替代流程**: 勾选"记住我"→ Token 有效期延长至 7 天
- **异常流程**: 连续 5 次登录失败 → 锁定 15 分钟，提示"账户已临时锁定，请 15 分钟后重试"

### UC-003: 发起新研究

- **角色**: 普通用户
- **前置条件**: 已登录，当前无进行中的研究
- **主流程**:
  1. 用户在主界面点击"新建研究"
  2. 输入研究主题（单行文本，支持自然语言描述）
  3. 选择研究模板类型（技术调研 / 竞品分析 / 论文综述 / 自定义）
  4. 点击"开始"
  5. 主 Agent 基于所选模板生成研究计划，展示在结构化工作台中
  6. 用户进入 UC-004
- **替代流程**:
  - 用户输入为空 → 提示"请输入研究主题"
  - 已有进行中的研究 → 提示"当前有一个进行中的研究，请等待完成或终止后再发起新研究"
- **异常流程**: LLM 生成计划超时（>30s）→ 提示"计划生成超时，请重试"

### UC-004: 审核与修改研究计划

- **角色**: 普通用户
- **前置条件**: 已完成 UC-003，研究计划已生成
- **主流程**:
  1. 工作台展示研究计划：每个 Sub-agent 的名称、研究目标、搜索方向
  2. 用户逐项审视计划内容
  3. 用户可在聊天面板中发送修改指令（如"增加一个对比竞品的子任务"、"去掉第 3 项"）
  4. 系统基于用户反馈重新生成研究计划
  5. 重复步骤 1-4，直到用户满意
  6. 用户点击"确认计划"
  7. 系统进入 UC-006
- **替代流程**:
  - 用户对初始计划满意 → 直接点击"确认计划"，跳过修改
  - 用户直接关闭页面 → 研究保存为草稿，后续可从历史列表恢复
- **异常流程**: 修改轮次过多（>10 轮）→ 提示"建议确认当前计划，或重新发起研究"

### UC-005: 中断研究

- **角色**: 普通用户
- **前置条件**: 研究正在执行中（UC-006）
- **主流程**:
  1. 用户点击工作台上的"停止"按钮
  2. 系统终止所有正在进行中的 Sub-agent
  3. 已完成的 Sub-agent 结果保留，未完成的标记为"已取消"
  4. 系统基于已有的部分结果生成部分报告
- **异常流程**:
  - 停止信号丢失 → 30 秒后强制超时终止
  - 用户在发起搜索后 10 秒内停止且无任何有效结果返回 → 直接终止，丢弃空任务，不生成报告

### UC-006: 监控研究执行进度

- **角色**: 普通用户
- **前置条件**: 研究计划已确认
- **主流程**:
  1. 工作台展示所有 Sub-agent 的执行状态卡片（待执行 / 执行中 / 已完成 / 失败）
  2. 每张卡片实时展示：当前步骤、已用时间、搜索轮次（1/2）
  3. Sub-agent 完成后卡片变绿，展示初步发现摘要
  4. 汇总 Agent 在所有 Sub-agent 完成后开始工作
  5. 最终报告渲染完成，跳转到 UC-007
- **替代流程**: 某个 Sub-agent 失败 → 卡片变红，展示失败原因，其余 Sub-agent 继续执行
- **异常流程**: 所有 Sub-agent 均失败 → 主 Agent 拒绝生成报告，直接终止本次研究并提示"所有搜索源均未返回有效信息，请检查 MCP Server 状态后重试"

### UC-007: 查看研究报告

- **角色**: 普通用户
- **前置条件**: 研究已完成（全成功或部分成功）
- **主流程**:
  1. 工作台展示三部分内容（Tab 切换）：
     - **研究计划**：原始确认的计划
     - **Sub-agent 结果**：每个 Sub-agent 的详细发现及原始来源
     - **研究汇总**：主 Agent 的综合报告（Markdown 渲染）
  2. 报告中所有引用可溯源——点击引用跳转到对应原始链接
  3. 用户可复制报告全文（纯文本/Markdown）
- **替代流程**: 用户对结果不满意 → 可点击"重新研究"，返回 UC-003 并预填当前主题
- **异常流程**: 报告渲染失败 → 降级展示原始 Markdown 文本

### UC-008: 浏览研究历史

- **角色**: 普通用户
- **前置条件**: 已登录
- **主流程**:
  1. 用户进入"历史记录"页面
  2. 列表展示所有历史研究，每条包含：主题、创建时间、状态（已完成/草稿/失败/已取消）、Token 消耗
  3. 支持按时间倒序排序（V1 固定按 created_at DESC，V2 可扩展 sortBy/sortOrder 参数）
  4. 点击某条记录 → 跳转到 UC-007 查看报告
- **异常流程**: 历史数据为空 → 展示空状态提示"还没有研究记录，去发起第一个研究吧"

### UC-009: 删除研究记录

- **角色**: 普通用户
- **前置条件**: 历史列表中存在记录
- **主流程**:
  1. 在历史列表中点击删除按钮
  2. 系统弹出二次确认
  3. 用户确认后，系统对记录执行软删除（标记 deleted_at，前端不再展示）
- **异常流程**: 删除过程中系统故障 → 数据回滚，保持原状

### UC-010: 恢复草稿研究

- **角色**: 普通用户
- **前置条件**: 存在 status='draft' 的研究
- **主流程**:
  1. 在历史列表中看到标记为"草稿"的研究
  2. 点击进入 → 工作台加载研究详情（GET /research/{id}）
  3. 根据 status='draft' 自动渲染计划修改面板
  4. 用户可继续修改计划并确认执行
- **替代流程**: 用户也可以删除草稿，不再恢复

### UC-011: 查看 Token 消耗仪表盘

- **角色**: 普通用户
- **前置条件**: 已登录，存在历史研究记录
- **主流程**:
  1. 用户在 Dashboard 页面查看 Token 消耗统计面板
  2. 展示：今日消耗、本周消耗、总研究次数、平均每次研究消耗
  3. 数据实时更新（每次研究完成后刷新）
- **异常流程**: 无历史数据 → 展示"暂无统计数据"占位提示

## 5. 非功能需求

| ID | 需求 | 指标 |
|---|---|---|
| NFR-001 | 研究计划生成耗时 | P95 < 15 秒 |
| NFR-002 | 完整研究链路耗时 | 目标 5-10 分钟（3-5 Sub-agent × 2 轮并行） |
| NFR-003 | SSE 推送延迟 | < 1 秒（状态变更到前端展示） |
| NFR-004 | API 响应时间（非 LLM 接口） | P99 < 200ms |
| NFR-005 | 数据库 | PostgreSQL，支持 ≥ 1 万用户的研究历史存储 |
| NFR-006 | 安全性 | 密码 bcrypt 不可逆存储；JWT 可验证 |
| NFR-007 | 部署方式 | Docker 自托管，单 `docker compose up` 启动 |
| NFR-008 | 并发研究数 | 单用户同时最多 1 个进行中的研究（防止 Token 消耗失控） |
| NFR-009 | LLM API 协议 | OpenAI Chat Completion 协议兼容 |
| NFR-010 | 报告最大规模 | 单次研究报告 ≤ 50,000 中文字符 |
| NFR-011 | Token 成本 | 单次研究基线 ≤ ¥5（待 POC-004 验证） |
| NFR-012 | Sub-agent 循环硬限制 | 每个 Sub-agent 最多 2 轮搜索循环 |
| NFR-013 | 主 Agent 拆分上限 | 每个研究计划 3-5 个 Sub-agent |

## 6. 里程碑

| 里程碑 | 内容 | 预计交付物 |
|---|---|---|
| M1 | 基础设施搭建 | 项目骨架、Docker Compose、PostgreSQL、Auth 模块（注册/登录/JWT） |
| M2 | 核心 Agent 编排 + 基础 MCP 连通 | 主 Agent 计划生成 → Sub-agent 单路执行（挂真实 MCP 搜索源）→ 汇总 → 报告输出 |
| M3 | 并行执行 + URL 去重 | Sub-agent 并行编排、URL 去重 |
| M4 | 结构化工作台 UI | 研究计划展示与多轮修改、实时进度面板、Markdown 报告渲染 |
| M5 | 历史与个人中心 | 研究历史列表、报告回溯、复制导出、Token 统计 |
| M6 | 联调与上线 | 全链路联调、POC 验证收尾、部署文档 |
